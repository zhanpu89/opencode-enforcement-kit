# 术语表

> 本文件定义 system-architect 技能中使用的专业术语和缩写，供 AI 在架构设计时保持术语一致性。

---

## 核心术语

| 术语 | 含义 | 使用场景 |
|-----|------|---------|
| **SAD** | 软件架构设计文档（Software Architecture Document） | Step 3 输出产物 |
| **PRD** | 产品需求文档（Product Requirements Document），prd-writer 技能的输出 | Step 0 输入源 |
| **技术栈** | 项目选用的框架、库、中间件、数据库等技术组合 | Step 2 技术选型 |
| **分层架构** | 按职责将系统分为表现层、业务层、数据层等多层结构 | Step 2 架构模式 |
| **模块** | 功能内聚的代码单元，通常对应一个独立的业务领域 | Step 2 模块划分 |
| **接口契约** | 前后端、微服务间的 API 定义规范（请求/响应结构、状态码） | Step 2 接口设计 |
| **ER 图** | 实体关系图（Entity-Relationship Diagram），描述数据库表结构 | Step 2 数据建模 |
| **技术债务** | 为快速交付而采用的临时方案，需在后续迭代中重构 | Step 4 风险评估 |
| **非功能需求** | 性能、安全、可用性等质量属性要求 | Step 1 需求澄清 |
| **架构决策记录（ADR）** | Architecture Decision Record，记录重要架构选择的理由和权衡 | 决策文档化 |

## 禁用词汇对照表

> 以下左列词汇禁止出现在 SAD 文档中，应替换为右列的规范表达。

| 禁用词汇 | 替代表达 |
|---------|---------|
| "用 XX 框架" | ✅ "采用 XX 框架，理由：[性能优势/团队熟悉度/生态成熟度]" |
| "数据库用 MySQL" | ✅ "持久化层选用 MySQL 8.0，支持事务和索引优化" |
| "前后端分离" | ✅ "采用前后端分离架构，前端通过 RESTful API 与后端通信" |
| "高可用" | ✅ "目标可用性 99.9%，通过主从复制 + 读写分离实现" |
| "微服务" | ✅ "按业务域拆分为 N 个独立服务，通过 API 网关统一入口" |
| "缓存" | ✅ "引入 Redis 缓存热点数据，TTL 设置为 5 分钟" |
| "消息队列" | ✅ "采用 RabbitMQ 实现异步解耦，消息持久化保证可靠性" |
| "容器化" | ✅ "使用 Docker 容器化部署，通过 Kubernetes 编排" |
| "后续优化" | ❌ 必须明确优化时机和具体方案 |
| "性能很好" | ❌ 必须给出量化指标（如"接口响应时间 < 200ms"） |

## 架构模式术语

| 模式 | 适用场景 | 关键特征 |
|-----|---------|---------|
| **单体架构** | 小型项目、团队 < 5 人 | 部署简单、开发快速 |
| **分层架构** | 中型项目、职责分离清晰 | 表现层/业务层/数据层 |
| **微服务架构** | 大型项目、团队 > 10 人 | 服务独立部署、技术栈异构 |
| **事件驱动架构** | 高并发、异步处理场景 | 消息队列、发布订阅 |
| **CQRS** | 读写分离、高并发读场景 | 命令查询职责分离 |
| **Serverless** | 流量波动大、按需计费 | 函数即服务（FaaS） |

## 技术选型术语

### 后端框架
| 技术 | 语言 | 适用场景 |
|-----|------|---------|
| Spring Boot | Java | 企业级应用、微服务 |
| Django / FastAPI | Python | 快速原型、AI 应用 |
| Express / NestJS | Node.js | 前端团队全栈、实时应用 |
| Gin / Echo | Go | 高性能 API、云原生 |

### 数据库
| 技术 | 类型 | 适用场景 |
|-----|------|---------|
| MySQL / PostgreSQL | 关系型 | 事务强一致性、复杂查询 |
| MongoDB | 文档型 | 灵活 schema、半结构化数据 |
| Redis | KV 缓存 | 热点数据缓存、分布式锁 |
| Elasticsearch | 搜索引擎 | 全文检索、日志分析 |

## 接口设计术语

| 术语 | 含义 |
|-----|------|
| **RESTful API** | 基于 HTTP 的资源导向接口设计风格 |
| **状态码** | HTTP 响应状态码（200 成功、400 客户端错误、500 服务端错误） |
| **幂等性** | 多次调用同一接口产生相同效果（GET/PUT/DELETE 应幂等） |
| **请求体** | HTTP POST/PUT 请求的 JSON/XML 数据负载 |
| **响应体** | HTTP 响应返回的 JSON/XML 数据结构 |
| **分页** | `{ page: 1, size: 20, total: 100, data: [...] }` 结构 |
| **错误码** | 业务层自定义错误编码（如 `E1001: 用户不存在`） |

## 文档命名规范

| 场景 | 文件名格式 | 示例 |
|-----|-----------|------|
| 架构设计文档 | `{项目名}_SAD.md` | `智能采集平台_SAD.md` |
| 技术栈清单 | `tech-stack.json` | `tech-stack.json`（统一命名，无中文名）|
| 接口设计文档 | `{项目名}_接口设计.md` | `智能采集平台_接口设计.md` |
| ER 图文件 | `{项目名}_ER图.png` 或 `.mermaid` | `智能采集平台_ER图.png` |
| 架构决策记录 | `ADR-{序号}-{标题}.md` | `ADR-001-选择PostgreSQL.md` |

## 质量属性术语

| 术语 | 定义 | 量化指标示例 |
|-----|------|------------|
| **可用性** | 系统正常运行时间占比 | 99.9%（年停机时间 < 8.76 小时） |
| **性能** | 响应时间和吞吐量 | API 响应 < 200ms，TPS > 1000 |
| **可扩展性** | 水平扩展能力 | 支持无状态扩容到 10 个实例 |
| **安全性** | 防护能力 | HTTPS + JWT 认证 + SQL 注入防护 |
| **可维护性** | 代码可读性和测试覆盖率 | 单元测试覆盖率 > 80% |
| **容错性** | 异常处理和降级策略 | 熔断阈值 50%，降级返回缓存数据 |
