## 六大评估维度

在做出最终选择前，从以下维度对每个技术候选项进行评估：

| 维度 | 关键问题 |
|------|---------|
| **成熟度与稳定性** | 是否已生产就绪？是否持续维护？社区是否活跃？是否有大规模部署案例？ |
| **性能** | 能否满足所需吞吐量/并发？延迟是否可接受？在预期数据量下表现如何？ |
| **可扩展性** | 是否支持水平扩展？垂直扩展上限？是否支持数据分区/分片？是否对无状态友好？ |
| **运维复杂度** | 部署和运维是否简单？可观测性工具是否成熟？灾难恢复方案是否完善？团队学习曲线如何？ |
| **成本** | 许可证费用（开源 vs 商业）？基础设施占用？是否有隐性成本（出口流量、按席收费）？ |
| **团队技术储备** | 团队是否有实战经验？学习曲线是否合理？文档和社区支持是否充分？ |

## 决策原则

1. **开源优先** —— 优先选择开源方案，避免厂商锁定，降低许可证成本。
2. **务实优于完美** —— 使用经过验证的成熟技术，避免过度设计。
3. **可落地的设计** —— 理论上更优但团队无法运维的选择是糟糕的选择。
4. **可扩展性与性能** —— 为增长而设计，在瓶颈出现前提前规划。
5. **安全内建** —— 优先选择安全记录良好、补丁活跃的技术。

## 记录选型理由

对于每个选定的技术，在设计文档和 `tech-stack.json` 中记录：
- **是什么** —— 简要描述
- **为什么选它** —— 关键原因（参考上方维度）
- **为什么不选其他** —— 考虑过的 1~2 个备选方案及排除原因
- **已知权衡** —— 对已接受的弱点进行客观评估

## 技术目录

### 后端框架

| 框架 | 语言 | 优势 | 劣势 | 最适合 |
|------|------|------|------|--------|
| Spring Boot | Java | 生态成熟、企业级、工具链完善 | 学习曲线陡、启动慢 | 企业应用、大型团队 |
| FastAPI | Python | 开发快、自动生成文档、异步支持好 | 生态比 Django 小 | 快速原型、API 服务 |
| Django | Python | 开箱即用、ORM 强大、社区大 | 性能较慢 | Web 应用、内容管理 |
| NestJS | Node.js | 结构化、企业级、TypeScript 优先 | 学习曲线较陡 | 企业应用、微服务 |
| Express | Node.js | 轻量灵活、高并发 | 结构松散、需自行组装 | API 服务、实时应用 |
| Gin | Go | 性能卓越、简洁 | 生态较小 | 高性能服务 |

**性能参考（请求/秒，近似值）：**
Go (Gin) 5万+ › Node.js (Express) 3万+ › Java (Spring Boot) 2万+ › Python (FastAPI) 1万+

### 数据库

| 数据库 | 类型 | 优势 | 劣势 | 最适合 |
|--------|------|------|------|--------|
| PostgreSQL | 关系型 | 功能完整、稳定、可扩展 | 水平扩展复杂 | 通用场景、复杂查询 |
| MySQL | 关系型 | 简单、快速、广泛采用 | 高级特性较少 | Web 应用、互联网级读取 |
| Redis | 键值型 | 极快、数据结构丰富 | 受内存限制 | 缓存、会话、实时数据 |
| MongoDB | 文档型 | Schema 灵活、水平扩展简单 | 事务支持弱 | 非结构化数据、快速迭代 |
| Elasticsearch | 搜索/分析 | 全文搜索、强大聚合 | 复杂度高、内存占用大 | 日志分析、全文搜索 |

### 消息队列

| 队列 | 优势 | 劣势 | 最适合 |
|------|------|------|--------|
| RabbitMQ | 功能完整、可靠、支持事务 | 复杂度较高 | 可靠消息传递、复杂工作流 |
| Kafka | 超高吞吐、持久化、流处理 | 复杂度高、学习曲线陡 | 日志收集、流处理、实时分析 |
| Redis Queue | 简单、快速 | 功能有限、可靠性较低 | 简单任务队列、后台作业 |

### 前端框架（Web）

| 框架 | 优势 | 劣势 | 最适合 |
|------|------|------|--------|
| Vue3 | 渐进式、学习曲线平缓、国内生态好 | 大型项目架构约束弱 | 中后台系统、国内团队 |
| React | 生态最大、灵活性高、Hooks 成熟 | 学习曲线较陡、需自行选型状态管理 | 大型应用、国际化团队 |

**推荐组合（Vue3）：** Vue3 + TypeScript + Vite + Element Plus + Pinia + Vue Router 4 + Axios

**推荐组合（React）：** React 18 + TypeScript + Vite + Ant Design + Zustand + React Router 6 + Axios

### 小程序框架

| 框架 | 优势 | 劣势 | 最适合 |
|------|------|------|--------|
| 微信原生 | 性能最好、官方支持 | 只能运行在微信 | 仅需微信小程序、性能敏感 |

## SAD 层协作边界

> 与 task-decomposer 协作时加载，明确两个技能的粒度分工，避免重复劳动。

| 章节 | system-architect（SAD 层）写到这里 | task-decomposer（详设层）从这里接手 |
|------|----------------------------------|----------------------------------|
| **数据库设计** | 核心表名 + 关键字段 + ER 关系描述 + 索引策略概述 | 完整 DDL（含所有字段、约束、索引）|
| **API 设计** | 端点列表 + HTTP 方法 + 功能描述 + 认证方案 | 完整 OpenAPI 3.0 Schema（含 Request/Response 字段定义）|
| **模块设计** | 模块划分 + 职责描述 + 模块间依赖关系 | 伪代码逻辑 + 算法细节 + 测试用例 |
| **前端路由/页面** | 页面清单 + 路由路径 + 权限要求（F2）；组件分层策略（F4） | 每个页面的交互规则 + API 调用映射 + 组件内部逻辑 |
| **小程序页面** | tabBar 结构 + 分包策略 + 微信能力集成方案 | 每个页面的生命周期钩子 + 缓存策略细节 + 错误处理代码 |

> **为什么这个边界很重要**：SAD 层写得太深（如完整 DDL、完整 OpenAPI Schema），task-decomposer 就是重复劳动；写得太浅，task-decomposer 就缺乏足够的输入。
