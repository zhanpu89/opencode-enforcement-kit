# 术语表

> 本文件定义 code-reviewer 技能中使用的专业术语和缩写，供 AI 在代码评审时保持术语一致性。

---

## 核心术语

| 术语 | 含义 | 使用场景 |
|-----|------|---------|
| **Code Review** | 代码评审，对代码质量、安全、性能、可维护性进行系统性检查 | 全流程核心活动 |
| **问题等级** | 🔴 严重 / 🟡 一般 / 🔵 建议 | 问题分类标准 |
| **代码异味** | Code Smell，代码中存在的潜在问题迹象 | 可维护性检查 |
| **技术债务** | Technical Debt，为快速交付而采用的临时方案，需后续重构 | 长期质量管理 |
| **安全漏洞** | Security Vulnerability，可被攻击利用的代码缺陷 | 安全性检查 |
| **性能瓶颈** | Performance Bottleneck，影响系统响应速度的代码段 | 性能检查 |
| **反模式** | Anti-pattern，常见的错误实践 | 负面示例 |
| **最佳实践** | Best Practice，经过验证的优秀实践 | 正面示例 |
| **Lint 错误** | 代码风格和潜在错误检查工具的报错 | 代码规范检查 |
| **测试覆盖率** | Test Coverage，代码被单元测试覆盖的比例 | 质量指标 |

## 禁用词汇对照表

> 以下左列词汇禁止出现在评审报告中，应替换为右列的规范表达。

| 禁用词汇 | 替代表达 |
|---------|---------|
| "代码写得不好" | ❌ 必须明确指出具体问题（如"UserService.java:45 缺少空指针检查"） |
| "建议优化" | ❌ 必须给出具体优化方案和代码示例 |
| "可能有 bug" | ❌ 必须确认问题是否存在，给出触发条件 |
| "不规范" | ✅ "违反命名规范：变量名 `a` 应改为 `userId`（驼峰命名）" |
| "有安全风险" | ✅ "存在 SQL 注入风险：第 X 行使用字符串拼接，应改用参数化查询" |
| "性能差" | ✅ "性能瓶颈：第 X 行循环内查询数据库，应改为批量查询" |
| "难以维护" | ✅ "函数过长（150 行），建议拆分为 3 个子函数：`validate`/`process`/`save`" |
| "全部通过" | ✅ "评审通过，无严重问题，X 个优化建议（见报告第 3 节）" |
| "重构一下" | ✅ "重构建议：将 `switch-case` 重构为策略模式" |
| "加个注释" | ✅ "缺少函数注释：`calculateDiscount` 应添加参数说明和返回值说明" |

## 问题等级定义

| 等级 | 标识 | 定义 | 处理要求 | 示例 |
|-----|------|------|---------|------|
| **严重** | 🔴 | 阻塞上线或存在安全漏洞的问题 | **必须修复** | SQL 注入、明文密码、NPE 崩溃、死循环 |
| **一般** | 🟡 | 影响代码质量或后续维护的问题 | **建议修复** | 命名不规范、函数过长、缺少异常处理、重复代码 |
| **建议** | 🔵 | 优化建议，不影响功能 | **可选修复** | 注释优化、变量名优化、日志补充 |

## 评审维度术语

| 维度 | 检查内容 | 合格标准 |
|-----|---------|---------|
| **功能正确性** | 代码逻辑是否正确实现需求 | 无逻辑错误、边界值处理正确 |
| **安全性** | 是否存在安全漏洞 | 无 SQL 注入、XSS、CSRF、明文密码等漏洞 |
| **性能** | 是否存在性能瓶颈 | 无 N+1 查询、无循环内 I/O、无内存泄漏 |
| **可读性** | 代码是否易于理解 | 命名清晰、注释充分、逻辑简洁 |
| **可维护性** | 代码是否易于扩展和修改 | 函数 < 50 行、类 < 500 行、低耦合高内聚 |
| **代码规范** | 是否符合团队编码规范 | 通过 ESLint / Pylint / golangci-lint 检查 |
| **测试覆盖** | 是否有充分的单元测试 | 核心业务逻辑测试覆盖率 > 80% |

## 代码异味（Code Smell）

| 异味类型 | 表现 | 重构建议 |
|---------|------|---------|
| **长函数** | 函数超过 50 行 | 拆分为多个子函数 |
| **长类** | 类超过 500 行 | 按职责拆分为多个类 |
| **重复代码** | 相同逻辑在多处出现 | 提取为公共函数/类 |
| **过长参数列表** | 函数参数 > 5 个 | 封装为对象参数 |
| **深层嵌套** | if-else 嵌套 > 3 层 | 提前 return、策略模式 |
| **魔法数字** | 硬编码的常量 | 定义为命名常量 |
| **注释过多** | 代码需要大量注释才能理解 | 重构代码使其自解释 |
| **无用代码** | 注释掉的代码、未使用的变量 | 删除 |

## 安全漏洞类型

| 漏洞类型 | 风险等级 | 典型场景 | 修复方案 |
|---------|---------|---------|---------|
| **SQL 注入** | 🔴 严重 | 字符串拼接 SQL | 使用参数化查询 / ORM |
| **XSS** | 🔴 严重 | 未转义用户输入直接渲染 | HTML 转义 / CSP |
| **明文密码** | 🔴 严重 | 密码明文存储或传输 | bcrypt 加密 + HTTPS |
| **硬编码密钥** | 🔴 严重 | 代码中直接写入 API Key | 使用环境变量 / 密钥管理 |
| **未授权访问** | 🔴 严重 | 接口缺少权限校验 | 添加认证中间件 |
| **敏感信息泄露** | 🟡 一般 | 日志记录用户密码 | 脱敏处理 |
| **CSRF** | 🟡 一般 | 状态变更接口未防 CSRF | 添加 CSRF Token |
| **路径遍历** | 🔴 严重 | 文件路径未校验 | 路径白名单校验 |

## 性能反模式

| 反模式 | 表现 | 影响 | 优化方案 |
|-------|------|------|---------|
| **N+1 查询** | 循环内查询数据库 | 数据库压力大 | 批量查询 / JOIN |
| **全表扫描** | 查询无索引字段 | 响应慢 | 添加索引 |
| **过度查询** | 查询全部字段但只用部分 | 网络带宽浪费 | 只查询需要的字段 |
| **同步阻塞** | 串行调用多个接口 | 响应慢 | 并发调用 / 异步 |
| **内存泄漏** | 对象未释放引用 | 内存溢出 | 及时释放 / 弱引用 |
| **大对象传输** | 接口返回巨大 JSON | 网络慢 | 分页 / 字段过滤 |
| **缺少缓存** | 频繁查询不变数据 | 数据库压力 | Redis 缓存热点数据 |

## 可维护性反模式

| 反模式 | 表现 | 影响 | 重构建议 |
|-------|------|------|---------|
| **上帝类** | 一个类承担过多职责 | 难以理解和修改 | 按职责拆分 |
| **意大利面代码** | 逻辑混乱、跳转频繁 | 难以调试 | 重新组织逻辑 |
| **紧耦合** | 类之间直接依赖具体实现 | 难以测试和扩展 | 依赖注入 / 接口抽象 |
| **缺少抽象** | 重复代码未提取 | 修改成本高 | 提取公共方法/类 |
| **过度设计** | 引入不必要的设计模式 | 复杂度高 | 简化设计 |
| **命名不清** | 变量名 `a`、`temp` | 难以理解 | 语义化命名 |

## 编码规范术语

| 规范类型 | 检查工具 | 常见规则 |
|---------|---------|---------|
| **JavaScript/TypeScript** | ESLint + Prettier | 2空格缩进、单引号、结尾分号 |
| **Python** | Pylint + Black | 4空格缩进、PEP 8 命名 |
| **Java** | Checkstyle + Google Java Style | 驼峰命名、2空格缩进 |
| **Go** | golangci-lint + gofmt | Tab 缩进、错误处理不忽略 |

## 测试术语

| 术语 | 说明 | 工具 |
|-----|------|------|
| **单元测试** | 函数级别的测试 | Jest / pytest / JUnit / go test |
| **集成测试** | 模块间协作的测试 | Supertest / pytest / TestRestTemplate |
| **测试覆盖率** | 代码被测试覆盖的比例 | Istanbul / coverage.py / JaCoCo |
| **Mock** | 模拟外部依赖（数据库、API） | jest.mock / unittest.mock / Mockito |
| **测试用例** | Given/When/Then 格式的测试场景 | 每个核心函数至少 3 个用例 |

## 评审流程术语

| 阶段 | 活动 | 产出 |
|-----|------|------|
| **Step 0** | 识别代码类型（后端/前端/语言） | 确定检查清单 |
| **Step 1** | 静态分析，运行 Lint 工具 | Lint 错误清单 |
| **Step 2** | 逐文件检查，对照检查清单 | 详细问题列表 |
| **Step 3** | 汇总问题，生成评审报告 | `{模块名}_代码评审报告.md` |
| **Step 4** | 给出修改建议和优先级 | 修改建议清单 |

---

## 评审报告术语

| 章节 | 内容 | 说明 |
|-----|------|------|
| **评审概要** | 模块名、提交 ID、评审日期、结论 | 一页纸总结 |
| **问题汇总** | 严重/一般/建议 问题数量统计 | 量化指标 |
| **详细问题清单** | 每个问题的等级/文件/行号/描述/建议 | 按等级分组 |
| **代码示例** | 问题代码 vs 修复后代码对比 | Diff 格式 |
| **修改建议** | 优先级排序的修复任务 | P0（必须改）/ P1（建议改）/ P2（可选） |
| **评审结论** | 通过 / 有条件通过 / 不通过 | 明确下一步行动 |

---

## 评审结论术语

| 结论 | 条件 | 下一步 |
|-----|------|-------|
| **通过** | 无严重问题，一般问题 < 3 个 | 合并代码 |
| **有条件通过** | 严重问题 ≤ 2 个，且有明确修复方案 | 修复严重问题后合并 |
| **不通过** | 严重问题 > 2 个，或存在安全漏洞 | 大幅修改后重新评审 |

## 常见问题模式

| 问题类型 | 典型代码 | 修复方案 |
|---------|---------|---------|
| **空指针异常（NPE）** | `user.getName()` 未判空 | 添加 `if (user != null)` 或使用 Optional |
| **资源未关闭** | 数据库连接未 close | 使用 try-with-resources |
| **并发问题** | 共享变量无同步 | 加锁 / 使用线程安全类 |
| **死循环** | `while(true)` 无退出条件 | 添加退出条件或超时 |
| **硬编码配置** | 端口号写死在代码中 | 抽取到配置文件 |
| **异常吞没** | 空 catch 块 | 记录日志或重新抛出 |
