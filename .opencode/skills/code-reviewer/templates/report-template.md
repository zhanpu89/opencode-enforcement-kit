## 七、P0 阻断项详情

> ⚠️ 以下问题**必须全部修复**后，才能进入测试阶段。

### CR-001：前后端字段名不一致（契约断裂）

- **位置：** `frontend/src/types/user.ts`，第 8 行
- **问题：** 前端类型定义 `username: string` 与后端 OpenAPI 定义 `userName: String` 字段名不一致（大小写差异），导致运行时数据为 `undefined`
- **风险等级：** P0 — 契约断裂，功能直接失效
- **修复方案：**

```typescript
// 修复前（错误）
export interface UserVO {
  username: string  // ❌ 与后端 userName 不一致
}

// 修复后（正确）
export interface UserVO {
  userName: string  // ✅ 与后端 OpenAPI 定义完全一致
}
```

---

### CR-002：SQL 注入漏洞

- **位置：** `src/main/resources/mapper/XxxMapper.xml`，第 23 行
- **问题：** `ORDER BY ${sortField}` 直接拼接用户输入，攻击者可构造恶意 SQL 执行任意数据库操作
- **风险等级：** P0 — 数据安全漏洞
- **修复方案：**

```java
// 在 Service 中调用 Mapper 前，先做白名单校验
private static final Set<String> ALLOWED_SORT_FIELDS = Set.of("create_time", "update_time", "name");

if (!ALLOWED_SORT_FIELDS.contains(sortField)) {
    throw new BizException(ErrorCode.PARAM_INVALID, "不支持的排序字段: " + sortField);
}
```

## 版本号变更规则

格式：`vMAJOR.MINOR.PATCH`

| 段位 | 触发条件 | 示例 |
|------|---------|------|
| **MAJOR** | 评审结论根本性变化 | v1.0.0 → v2.0.0 |
| **MINOR** | 新增评审维度或重大补充 | v1.0.0 → v1.1.0 |
| **PATCH** | 笔误修正、补充遗漏检查项 | v1.0.0 → v1.0.1 |

**多轮评审规则：** 代码修复后重新评审时，**新建文件**并在文件名加 `_R2`、`_R3` 后缀，版本重置为 `v1.0.0`。
