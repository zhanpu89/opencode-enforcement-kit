# 术语表

> 本文件定义 review-expert 技能中使用的专业术语和缩写，供 AI 在文档评审时保持术语一致性。

---

## 核心术语

| 术语 | 含义 | 使用场景 |
|-----|------|---------|
| **评审** | Review，对文档质量、完整性、合理性进行系统性检查 | 全流程核心活动 |
| **PRD** | 产品需求文档（Product Requirements Document） | 评审对象类型 |
| **SAD** | 软件架构设计文档（Software Architecture Document） | 评审对象类型 |
| **规则表** | `doc/detailed/项目规则.md`，记录全局约定和契约 | 评审对象类型 |
| **设计明细表** | 后端/前端设计明细的结构化表格 | 评审对象类型 |
| **检查清单** | Checklist，按文档类型划分的检查项列表 | Step 2 检查依据 |
| **问题等级** | 🔴 严重 / 🟡 一般 / 🔵 建议 | 问题分类标准 |
| **可执行性** | 开发人员能否基于文档直接实施，无需反复询问 | 质量核心指标 |
| **一致性** | 文档内部术语、命名、风格的统一性 | 质量核心指标 |
| **完整性** | 所有必要信息是否齐全（功能点/接口/字段/逻辑） | 质量核心指标 |

## 禁用词汇对照表

> 以下左列词汇禁止出现在评审报告中，应替换为右列的规范表达。

| 禁用词汇 | 替代表达 |
|---------|---------|
| "文档写得不好" | ❌ 必须明确指出具体问题（如"第 3.2 节缺少接口响应格式"） |
| "建议优化" | ❌ 必须给出具体优化方案 |
| "可能有问题" | ❌ 必须确认问题是否存在，给出明确结论 |
| "需要完善" | ❌ 必须列出缺失内容和补充位置 |
| "表述不清" | ✅ "第 X 节【标题】表述不清，建议改为：[具体文字]" |
| "不合理" | ✅ "技术选型不合理，理由：[性能瓶颈/成本过高/团队不熟悉]" |
| "有遗漏" | ✅ "功能点遗漏：PRD 第 X 节的【功能名】在设计明细表中未体现" |
| "格式错误" | ✅ "表格格式错误：第 X 行缺少【字段名】列" |
| "不一致" | ✅ "术语不一致：第 2 节用'用户'，第 5 节用'会员'，建议统一为'用户'" |
| "全部通过" | ✅ "评审通过，无严重问题，X 个建议项（见报告第 3 节）" |

## 问题等级定义

| 等级 | 标识 | 定义 | 处理要求 | 示例 |
|-----|------|------|---------|------|
| **严重** | 🔴 | 阻塞开发或导致系统无法运行的问题 | **必须修复** | 核心接口缺失、数据库表未设计、技术栈冲突 |
| **一般** | 🟡 | 影响开发效率或后续维护的问题 | **建议修复** | 命名不规范、字段说明不清、缺少错误处理 |
| **建议** | 🔵 | 优化建议，不影响功能实现 | **可选修复** | 文档排版、示例代码补充、注释优化 |

## 评审维度术语

| 维度 | 检查内容 | 合格标准 |
|-----|---------|---------|
| **完整性** | 所有必要信息是否齐全 | PRD 功能点 100% 覆盖到设计明细表 |
| **准确性** | 技术选型、接口设计是否正确 | 无技术错误、无逻辑矛盾 |
| **一致性** | 术语、命名、风格是否统一 | 同一概念在全文使用同一术语 |
| **可行性** | 技术方案是否可实施 | 技术栈成熟、团队有能力实现 |
| **可维护性** | 架构是否易于扩展和维护 | 模块解耦、代码可读性高 |
| **合规性** | 是否符合项目规则表约定 | 遵守 LC-* 规则 |

## 文档类型评审重点

### PRD 评审重点
| 检查项 | 说明 |
|-------|------|
| **业务目标** | 是否明确"为什么做"和"解决什么问题" |
| **用户故事** | 是否覆盖所有用户角色和使用场景 |
| **验收标准（AC）** | 每个功能是否有 Given/When/Then 格式的 AC |
| **范围边界** | 是否明确"做什么"和"不做什么" |
| **无技术细节** | PRD 应只描述业务逻辑，不涉及技术实现 |

### SAD 评审重点
| 检查项 | 说明 |
|-------|------|
| **技术栈** | 是否明确框架、数据库、中间件及版本 |
| **架构模式** | 是否明确分层结构和模块划分 |
| **接口设计** | 是否定义前后端接口契约 |
| **数据建模** | 是否包含 ER 图和表结构说明 |
| **非功能需求** | 是否明确性能、安全、可用性指标 |

### 规则表评审重点
| 检查项 | 说明 |
|-------|------|
| **规则编号** | 是否遵循 LC-{类别}-{序号} 格式 |
| **规则完整性** | 是否覆盖技术栈、目录结构、命名规范、错误处理 |
| **规则可执行** | 是否给出具体示例和反例 |
| **规则一致性** | 多个规则之间是否有冲突 |

### 设计明细表评审重点
| 检查项 | 说明 |
|-------|------|
| **任务覆盖** | 是否覆盖 PRD 所有功能点 |
| **接口完整** | 后端接口是否包含路径/请求/响应/逻辑/数据库操作 |
| **页面完整** | 前端页面是否包含路由/组件/接口/字段/交互 |
| **依赖清晰** | 任务间依赖关系是否明确 |
| **粒度适中** | 任务工作量是否为 0.5-1 天 |

## 评审流程术语

| 阶段 | 活动 | 产出 |
|-----|------|------|
| **Step 0** | 识别文档类型（PRD/SAD/规则表/设计明细表） | 确定检查清单 |
| **Step 1** | 快速浏览，了解整体结构 | 结构性问题列表 |
| **Step 2** | 逐节检查，对照检查清单 | 详细问题列表（含等级和位置） |
| **Step 3** | 汇总问题，生成评审报告 | `{文档名}_评审报告.md` |
| **Step 4** | 给出修改建议和优先级 | 修改建议清单 |

---

## 评审报告术语

| 章节 | 内容 | 说明 |
|-----|------|------|
| **评审概要** | 文档名称、版本、评审日期、结论 | 一页纸总结 |
| **问题汇总** | 严重/一般/建议 问题数量统计 | 量化指标 |
| **详细问题清单** | 每个问题的等级/位置/描述/建议 | 按等级分组 |
| **修改建议** | 优先级排序的修改任务清单 | P0（必须改）/ P1（建议改）/ P2（可选） |
| **评审结论** | 通过 / 有条件通过 / 不通过 | 明确下一步行动 |

---

## 评审结论术语

| 结论 | 条件 | 下一步 |
|-----|------|-------|
| **通过** | 无严重问题，一般问题 < 3 个 | 进入开发阶段 |
| **有条件通过** | 严重问题 ≤ 2 个，且有明确修复方案 | 修复严重问题后进入开发 |
| **不通过** | 严重问题 > 2 个，或缺少核心章节 | 大幅修改后重新评审 |

## 文档命名规范

| 场景 | 文件名格式 | 示例 |
|-----|-----------|------|
| 评审报告 | `{文档名}_评审报告.md` | `智能采集平台_PRD_评审报告.md` |
| 问题清单 | `{文档名}_问题清单.xlsx` | `智能采集平台_SAD_问题清单.xlsx` |
| 修改版本 | `{文档名}_v{版本号}.md` | `智能采集平台_PRD_v2.0.md` |

## 常见问题模式

| 问题类型 | 典型表现 | 修复建议 |
|---------|---------|---------|
| **功能遗漏** | PRD 有功能，设计明细表无任务 | 补充对应接口/页面任务 |
| **接口不完整** | 只有路径，无请求/响应格式 | 补充请求体、响应体、状态码 |
| **术语不统一** | 同一概念多种叫法 | 建立术语表，全文替换 |
| **逻辑矛盾** | 第 3 节说"需要登录"，第 5 节说"匿名访问" | 明确访问权限，修正矛盾 |
| **技术栈过时** | 使用淘汰版本（如 Vue 2 / Python 2） | 升级到现代版本 |
| **缺少异常处理** | 只写正常流程，无错误处理 | 补充异常场景和错误码 |
