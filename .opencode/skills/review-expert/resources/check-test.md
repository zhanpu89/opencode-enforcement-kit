# 测试用例评审检查清单

> Step 2 测试用例评审时加载。包含需求覆盖度、场景深度、逆向场景、用例质量、数据与环境策略、自动化分级检查。

## 四、测试用例评审检查清单

### 4.1 需求覆盖度检查

- [ ] **AC双向追溯**：每条AC是否有对应的测试用例？每个测试用例是否能追溯到AC？
- [ ] **功能点覆盖**：PRD中的每个功能点是否都有测试用例？
- [ ] **业务规则覆盖**：详细设计中的每条业务规则是否有对应测试用例？
- [ ] **接口覆盖**：每个API接口是否有测试用例（包括正常和异常响应）？

**覆盖率计算方法：**
```
需求覆盖率 = 有对应测试用例的AC数 / 总AC数 × 100%
目标：需求覆盖率 ≥ 100%（所有AC必须有用例）
```

---

### 4.2 场景深度检查

| 测试类型 | 最低占比要求 | 检查项 |
|---------|-----------|-------|
| 功能测试（正向） | 30-40% | 主流程是否完整覆盖 |
| 异常/容错测试 | ≥ 20% | 错误输入、服务不可用、超时 |
| 边界值测试 | ≥ 15% | 最小值、最大值、边界±1 |
| 性能/压力测试 | ≥ 5% | 并发量、响应时间、吞吐量 |
| 安全测试 | ≥ 5% | 越权、注入、认证绕过 |

- [ ] **容错测试**：网络超时、服务宕机、数据库不可用等场景是否有用例？
- [ ] **边界测试**：每个有约束的字段是否有边界值测试（最小值、最大值、超出边界）？
- [ ] **压力测试**：是否有对应架构评审中定义的并发指标的压力测试用例？
- [ ] **安全测试**：是否有越权访问、SQL注入、XSS等安全测试用例？

---

### 4.3 逆向场景检查

- [ ] **取消/撤销操作**：是否有取消、撤销、回退操作的测试用例？
- [ ] **超时场景**：操作超时后系统状态是否有测试用例？
- [ ] **重试场景**：重复提交、重复调用的幂等性是否有测试用例？
- [ ] **降级场景**：服务降级后的用户体验是否有测试用例？
- [ ] **并发冲突**：多用户同时操作同一资源的冲突场景是否有测试用例？

---

### 4.4 用例质量检查

- [ ] **用例独立性**：每个用例是否独立，不依赖其他用例的执行顺序？
- [ ] **用例原子性**：每个用例是否只验证一个行为？
- [ ] **输入完整**：每个用例的输入条件是否完整且具体（有实际数据值）？
- [ ] **预期明确**：每个用例的预期结果是否明确且可验证？
- [ ] **无重复用例**：是否存在完全相同或高度相似的重复用例？
- [ ] **命名规范**：用例ID和描述是否符合规范，能让人立即理解测试意图？

---

### 4.5 测试数据与环境策略检查

- [ ] **测试数据策略**：测试数据如何准备？是否有数据初始化脚本？
- [ ] **数据隔离**：测试用例之间的数据是否相互隔离？
- [ ] **Mock策略**：外部依赖（第三方API、消息队列）的Mock策略是否明确？
- [ ] **Mock与架构一致**：Mock的行为是否与架构设计中定义的接口契约一致？
- [ ] **环境依赖**：测试环境的依赖组件（数据库、Redis、MQ）是否可用？
- [ ] **数据清理**：测试执行后，测试数据是否会被清理？

---

### 4.6 自动化分级检查

- [ ] **自动化候选识别**：哪些用例适合自动化（稳定、高频、可重复）？
- [ ] **探索性测试识别**：哪些用例适合探索性测试（UI交互、用户体验）？
- [ ] **自动化优先级**：核心业务流程的自动化用例是否优先实现？
- [ ] **维护成本评估**：自动化用例的维护成本是否在可接受范围内？
