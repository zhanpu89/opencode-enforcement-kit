# 架构评审检查清单

> Step 2 架构评审时加载。包含系统边界、非功能属性、单点故障、技术选型、数据一致性、安全性检查。

## 二、架构评审检查清单

### 2.1 系统边界与模块划分

- [ ] **模块职责清晰**：每个模块的职责是否单一且明确？
- [ ] **接口契约定义**：模块间的接口是否有明确的契约定义（输入/输出/错误码）？
- [ ] **上下游关系**：系统与外部依赖的交互方式是否明确（同步/异步/事件驱动）？
- [ ] **数据所有权**：每份数据由哪个模块负责写入和维护？是否有数据所有权冲突？

---

### 2.2 非功能属性检查

> 🏷️ 标注说明：`[全阶段]` = MVP 和生产版均检查；`[生产版]` = MVP 阶段自动降级为 P2，不阻断。

| 非功能属性 | 检查项 | 是否量化 | 适用阶段 |
|----------|-------|---------|---------|
| **高可用** | SLA目标（如99.9%）是否定义？ | □ | `[生产版]` |
| **并发能力** | 峰值TPS/QPS是否估算？ | □ | `[生产版]` |
| **响应时间** | P99/P95延迟目标是否定义？ | □ | `[生产版]` |
| **数据一致性** | 一致性级别（强一致/最终一致）是否明确？ | □ | `[全阶段]` |
| **安全等级** | 认证方式、权限模型是否定义？ | □ | `[全阶段]` |
| **可扩展性** | 水平扩展策略是否描述？ | □ | `[生产版]` |
| **容灾能力** | RTO/RPO目标是否定义？ | □ | `[生产版]` |
| **存储容量** | 数据增长预估是否有？ | □ | `[生产版]` |

---

### 2.3 单点故障分析

对每个关键组件执行以下检查：

- [ ] **数据库**：是否有主从复制？主库宕机时如何切换？切换期间数据会丢失多少？
- [ ] **缓存层**：Redis宕机时，系统是否能降级运行？会有雪崩风险吗？
- [ ] **消息队列**：MQ宕机时，消息会丢失吗？消费者重启后能否继续消费？
- [ ] **外部服务**：第三方API不可用时，是否有熔断/降级/重试策略？
- [ ] **定时任务**：定时任务执行时间超过间隔时，会并发执行吗？
- [ ] **文件存储**：OSS/MinIO不可用时，上传/下载功能如何降级？

**What-If 追问清单：**
> - "如果这个核心服务挂了，依赖它的N个下游会发生什么？"
> - "如果流量突增10倍，哪个节点会先崩溃？"
> - "如果数据库主库宕机，数据会丢失多少？"
> - "如果消息积压了100万条，系统会怎样？"

---

### 2.4 技术选型评审

对每个关键技术选型执行以下检查：

- [ ] **选型依据**：是否说明了为什么选这个技术而不是替代方案？
- [ ] **替代方案对比**：是否列出了至少1个替代方案并说明了不选的原因？
- [ ] **团队熟悉度**：团队是否有使用该技术的经验？学习成本是否可接受？
- [ ] **社区活跃度**：该技术是否有活跃的社区支持？是否有长期维护保障？
- [ ] **许可证合规**：开源组件的许可证是否与商业使用兼容？

---

### 2.5 数据一致性检查

- [ ] **分布式事务**：跨服务的数据操作是否有一致性保障方案（Saga/TCC/最终一致）？
- [ ] **缓存一致性**：数据库更新后，缓存如何同步？是否有缓存击穿/穿透/雪崩的防护？
- [ ] **消息幂等**：消息消费者是否具备幂等性？重复消费会产生副作用吗？
- [ ] **数据迁移**：如果需要数据迁移，是否有回滚方案？

---

### 2.6 安全性检查

- [ ] **认证机制**：是否定义了认证方式（JWT/OAuth/Session）？Token如何失效？
- [ ] **授权模型**：RBAC/ABAC是否明确？每个接口的权限要求是否定义？
- [ ] **数据加密**：敏感数据（手机号、身份证、密码）是否加密存储？
- [ ] **传输安全**：是否强制HTTPS？是否有防重放攻击机制？
- [ ] **注入防护**：是否有SQL注入、XSS、CSRF的防护措施？
- [ ] **限流防护**：是否有接口限流？是否有防暴力破解机制？
