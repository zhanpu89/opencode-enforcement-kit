# 详细设计评审检查清单（含跨端对齐）

> Step 2 详细设计/多端详设评审时加载。包含并发安全、事务边界、幂等设计、异常处理、数据模型、可观测性、功能链完整性、跨端接口对齐检查。

## 三、详细设计评审检查清单

### 3.1 并发安全检查

- [ ] **共享资源识别**：文档中是否识别了所有可能被并发访问的共享资源？
- [ ] **锁机制选择**：使用乐观锁还是悲观锁？选择依据是否合理？
- [ ] **锁粒度**：锁的粒度是否合适？是否存在锁范围过大导致性能问题？
- [ ] **死锁预防**：多个锁的获取顺序是否一致？是否有死锁风险？
- [ ] **重试策略**：乐观锁冲突后的重试次数和间隔是否定义？超过重试次数怎么办？
- [ ] **分布式锁**：如果使用Redis分布式锁，锁超时时间是否合理？锁续期是否考虑？

**What-If 追问清单：**
> - "如果100个用户同时抢购同一件商品，库存会超卖吗？"
> - "如果乐观锁重试了3次都失败，用户看到什么？"
> - "如果持有锁的服务实例宕机，锁会自动释放吗？"

---

### 3.2 事务边界检查

- [ ] **事务范围合理**：每个事务是否只包含必要的操作？是否存在大事务？
- [ ] **跨库事务**：是否有跨数据库的事务操作？如何保证一致性？
- [ ] **事务与外部调用**：事务内是否有外部HTTP调用？（这是反模式，会导致长事务）
- [ ] **回滚策略**：事务失败时，哪些操作会回滚？哪些不会（如：已发送的短信）？
- [ ] **补偿机制**：对于无法回滚的操作（如：第三方扣款），是否有补偿方案？

---

### 3.3 幂等设计检查

- [ ] **写操作幂等**：所有写操作（创建/更新/删除）是否具备幂等性？
- [ ] **幂等机制**：使用什么机制保证幂等（唯一索引/幂等表/Token机制）？
- [ ] **幂等键定义**：幂等键的生成规则是否明确？客户端还是服务端生成？
- [ ] **幂等有效期**：幂等键的有效期是多长？过期后重复请求如何处理？
- [ ] **消息幂等**：消息消费者是否有幂等保护？

---

### 3.4 异常处理检查

- [ ] **异常分类**：是否区分了业务异常和系统异常？处理方式是否不同？
- [ ] **异常传播**：异常是否在正确的层级被捕获和处理？是否有异常被吞掉？
- [ ] **错误码定义**：每种异常是否有对应的错误码和用户友好的错误信息？
- [ ] **重试策略**：哪些操作失败后可以重试？重试次数和间隔是否定义？
- [ ] **死信处理**：消息消费失败超过重试次数后，进入死信队列，如何处理？
- [ ] **降级策略**：核心功能不可用时，是否有降级方案？降级后用户体验如何？

---

### 3.5 数据模型检查

- [ ] **索引覆盖**：常用查询条件是否有对应索引？是否有全表扫描风险？
- [ ] **索引冗余**：是否存在重复或低效的索引？
- [ ] **字段类型**：字段类型是否合适？是否有用VARCHAR存数字、用TEXT存短字符串的情况？
- [ ] **字段长度**：字段长度是否合理？是否有过长或过短的情况？
- [ ] **NULL处理**：哪些字段允许NULL？NULL值的业务含义是否明确？
- [ ] **软删除**：是否使用软删除？软删除字段是否在索引中考虑？
- [ ] **时间字段**：是否有create_time和update_time？时区处理是否一致？
- [ ] **大字段处理**：TEXT/BLOB等大字段是否会影响查询性能？是否考虑分离存储？

---

### 3.6 可观测性检查

- [ ] **关键操作日志**：核心业务操作是否有INFO级别日志（包含关键参数）？
- [ ] **异常日志**：所有异常是否有ERROR级别日志（包含堆栈信息）？
- [ ] **性能日志**：慢操作（>1s）是否有WARN级别日志？
- [ ] **日志脱敏**：日志中是否有手机号、密码等敏感信息？是否脱敏处理？
- [ ] **链路追踪**：是否有TraceId贯穿整个请求链路？
- [ ] **监控指标**：是否定义了关键业务指标（如：订单成功率、支付成功率）？
- [ ] **告警规则**：是否定义了告警阈值和通知方式？

---

### 3.7 功能链完整性检查

这是详细设计评审中最容易被忽视的维度。设计文档往往只描述了"显式需求"的接口，但每个写操作背后都有一组隐含的支撑接口。评审时需要系统性地检查这6类断链场景：

**3.7.1 操作前置依赖**

| 检查项 | 检查方式 | 风险等级 |
|--------|---------|---------|
| 有 `PUT /{resource}/{id}` 是否有 `GET /{resource}/{id}` | 遍历接口清单，检查每个写操作是否有对应读操作 | P1 |
| 有 `DELETE /{resource}/{id}` 是否有 `GET /{resource}/{id}` | 同上 | P1 |
| 有 `PATCH /{resource}/{id}/status` 是否有 `GET /{resource}/{id}` | 同上 | P1 |
| 有子资源操作是否有父资源 `GET /{parent}/{id}` | 检查子资源接口的父资源查询 | P1 |
| 有批量操作是否有对应列表查询接口 | 检查批量操作的数据来源 | P2 |

**3.7.2 状态机完整性**

- [ ] **状态枚举定义**：模块中有状态字段（`status`/`state`/`phase`）时，是否有完整的状态枚举定义？
- [ ] **合法转换表**：是否明确定义了哪些状态可以转换到哪些状态？（防止非法状态跳转）
- [ ] **前置状态校验**：每个状态变更接口的伪代码中，是否有"校验当前状态是否允许此次变更"的步骤？
- [ ] **逆向路径**：有审批/驳回流程时，是否有"驳回后重新提交"的接口？
- [ ] **并发保护**：并发状态变更场景是否有乐观锁（`version` 字段）或分布式锁说明？

**What-If 追问：**
> - "如果订单已经是'已完成'状态，用户能直接调用'取消'接口吗？伪代码里有校验吗？"
> - "审批被驳回后，申请人如何修改并重新提交？有对应接口吗？"

**3.7.3 跨模块依赖完整性**

- [ ] **依赖接口存在性**：第13节（依赖关系）中列出的每个外部接口路径，在被依赖模块的设计文档中是否真实存在？
- [ ] **接口契约一致性**：依赖方期望的请求/响应格式，与被依赖方定义的接口契约是否一致？
- [ ] **无悬空依赖**：是否存在"依赖了某个接口，但该接口在任何模块的设计文档中都找不到"的情况？
- [ ] **请求字段对齐**：依赖方传入的请求字段（字段名、类型、是否必填）与被依赖方接口定义的入参是否逐字段对齐？
- [ ] **响应字段对齐**：依赖方期望使用的响应字段（字段名、类型、层级路径）是否在被依赖方的响应体定义中真实存在？
- [ ] **枚举值对齐**：依赖方传入或使用的枚举值是否与被依赖方定义的枚举范围完全一致（无多无少）？

> ⚠️ **重点提示**：接口路径存在并不等于契约对齐。字段层级的不对齐（字段名拼写差异、类型不匹配、必填/可选不一致）是前后端联调阶段最高频的缺陷来源，此3项检查在详设阶段即可拦截。

**3.7.4 数据生命周期完整性**

- [ ] **创建有删除**：有创建附件/关联关系的接口，是否有对应的删除接口？
- [ ] **绑定有解绑**：有绑定关系的接口，是否有对应的解绑接口？
- [ ] **草稿可查询**：有保存草稿的接口，是否有草稿列表查询接口？
- [ ] **异步有结果查询**：有批量导入/导出的接口，是否有对应的任务状态查询接口？
- [ ] **软删除有恢复**：使用软删除（`deleted`）时，是否考虑了数据恢复接口（视业务需要）？

**3.7.5 异步流程完整性**

- [ ] **异步任务有状态查询**：触发异步任务的接口，是否有对应的 `GET /tasks/{id}/status` 查询接口？
- [ ] **MQ消费有死信处理**：消息消费逻辑是否定义了消费失败超过重试次数后的死信处理方案？
- [ ] **回调有签名验证**：接收第三方回调的接口，伪代码中是否有签名验证步骤？

**3.7.6 权限与数据隔离完整性**

- [ ] **多租户过滤**：多租户系统的列表查询接口，伪代码中是否有 `tenant_id` 过滤条件？
- [ ] **数据归属校验**：有数据归属设计时，`update`/`delete` 接口的伪代码中是否有 `ownerId == currentUserId` 校验？
- [ ] **角色权限注解**：管理员接口是否有 `@PreAuthorize` 或等效的权限校验步骤？

**3.7.7 上游需求/架构 SAD 正向比对（新增，有上游文档时必须执行）**

> 以上 3.7.1~3.7.6 均是在详设文档内部检查"写操作是否有配套读操作"等横向断链。  
> 本节检查**纵向断链**：上游文档（PRD 或架构 SAD）中明确规划的接口，在本模块详设文档中是否真实定义——这正是"需求有要求但后端直接遗漏"的根本来源。

**执行前提：** 评审时如能访问到 PRD（`doc/prd/`）或架构 SAD（`doc/arch/`），则本节为**强制执行**；上游文档均不可访问时，标注"跳过：上游文档不可访问"。

**执行步骤：**

1. **提取上游接口期望清单**：
   - 扫描 `doc/arch/` 中与本模块相关的 SAD（前端/小程序/后端均检查）或 `doc/prd/` 中的功能清单，提取所有明确提及的接口路径/功能点
   - 若文档中有接口列表章节（如"对外暴露接口"、"API 清单"、"功能列表"），逐行提取；没有则从用户故事/AC 中推导

2. **与本模块详设第3节接口列表做正向比对**：
   - 对每条上游期望接口，检查详设文档第3节 OpenAPI 定义中是否存在对应实现
   - **不存在 → 直接定级 P0（接口缺失：上游有需求，详设无定义）**
   - 输出格式：`❌ 接口缺失：{HTTP方法} {路径}（来源：{上游文档名} § {章节}）`

**What-If 追问：**
> - "PRD 里的功能 F，在这份详设第3节里能找到对应接口吗？"
> - "架构 SAD 里前端/小程序调用的接口列表，在这份详设里是否全部有定义？"
> - "有没有哪个上游要求的接口，在详设里连影子都找不到？"

---

**功能链断链的风险等级判定：**

| 断链类型 | 风险等级 | 判定理由 |
|---------|---------|---------|
| 有 `update` 无 `getById` | P1 | 前端表单无法回显，功能实际不可用 |
| 状态变更无前置校验 | P1 | 可能产生非法状态，数据一致性风险 |
| 跨模块依赖接口不存在 | P0 | 编码阶段必然报错，阻断开发 |
| 有绑定无解绑 | P2 | 功能不完整，但不阻断核心流程 |
| 异步任务无状态查询 | P2 | 用户体验差，但不影响数据正确性 |
| 多租户查询无隔离过滤 | P0 | 数据安全漏洞，越权访问风险 |
| **上游有需求但详设无对应接口定义（3.7.7）** | **P0** | **需求断链，编码阶段必然遗漏该功能** |
| **某端有接口定义但其他端无对应接口（3.8.2）** | **P0** | **跨端存在性断链，联调阶段必然报404** |
| **接口存在但请求字段名/类型/必填不对齐（3.8.3）** | **P1** | **联调阶段字段解析失败，功能不可用** |
| **接口存在但响应字段名/层级不对齐（3.8.3）** | **P1** | **前端/小程序无法正确解析响应体** |
| **枚举值范围不一致（3.8.3）** | **P1** | **枚举越界或无法识别，业务逻辑分支失效** |
| **字段命名风格差异（驼峰/下划线）（3.8.3）** | **P2** | **需序列化层统一转换，增加维护成本** |

## 三·八、跨端接口对齐检查清单（多端详设评审时执行）

> **适用场景**：仅当 Step 0 识别的评审模式为 `多端详设评审`（用户同时提供 ≥2 端的详细设计文档）时执行本节检查。单端 `详细设计评审` 跳过本节。
>
> **端的定义**：前端（Web）、后端（Server）、小程序（MiniApp）均视为独立的「端」，每两端之间的接口调用关系均需执行本节检查。
>
> **检查顺序**：先执行 3.8.1 端清单确认 → 3.8.2 存在性比对 → 3.8.3 字段层级对齐，三步缺一不可。

---

### 3.8.1 被评审端清单确认（前置，必须执行）

在执行任何对齐检查之前，先明确本次涉及的端：

- [ ] **端清单完整**：已确认本次涉及的所有端（前端/后端/小程序），无遗漏
- [ ] **调用关系明确**：已确认哪些端之间存在接口调用关系（如：前端→后端，小程序→后端）
- [ ] **接口清单可提取**：每端的详设文档中均包含接口清单章节，可供提取

> 🔴 **若某端文档缺少接口清单章节**，标记该端"接口清单不可提取"，跳过该端的跨端检查，并在报告注明。

**执行步骤**：读取各端详设接口清单章节，构建各端接口清单（格式：`HTTP方法 路径`），输出确认摘要：
```
📋 跨端对齐检查范围：
- 前端：X 个接口
- 后端：Y 个接口
- 小程序：Z 个接口
调用关系：前端→后端，小程序→后端
```

---

### 3.8.2 接口存在性比对（每对调用关系均执行）

> **核心问题**：调用方（前端/小程序）引用的每个接口，被调用方（后端）是否真实定义了？

- [ ] **正向比对（调用方→被调用方）**：调用方依赖的每个接口路径，在被调用方接口清单中均能找到对应定义（HTTP方法+路径完全匹配）
- [ ] **反向确认（被调用方→调用方）**：被调用方定义的接口，每个均有调用方使用；发现"孤岛接口"须标注并询问用户是否为预留接口（不强制P0）

**输出格式（发现问题时）**：
```
❌ [前端→后端] 接口缺失（P0）：
   - POST /api/order/submit（前端详设 §3.2 引用，后端详设中未定义）
   - GET /api/user/profile/{id}（前端详设 §2.1 引用，后端详设中未定义）
⚠️ [后端] 孤岛接口（待确认）：
   - DELETE /api/order/{id}（后端已定义，前端/小程序均未见调用，是预留接口吗？）
```

**What-If 追问：**
> - "前端详设里调用的 `POST /api/xxx`，在后端详设的接口列表里能找到吗？"
> - "小程序调用的每一个接口，后端都定义了吗？有没有后端根本没定义但小程序已经在调用的？"

---

### 3.8.3 字段层级对齐检查（对 3.8.2 通过的每个接口逐一执行）

> **核心问题**：接口路径一致不代表"能用"——字段名、类型、必填性、枚举值的任何一项不对齐都会导致联调失败。

**A. 请求参数字段对齐**

- [ ] **字段名对齐**：调用方传入的每个字段名（含嵌套子字段），与被调用方入参字段名完全一致（区分大小写、驼峰/下划线）
- [ ] **字段类型对齐**：每个字段的数据类型一致（`string` vs `number`、`timestamp` vs `datetime` 等）
- [ ] **必填性对齐**：调用方标注可选的字段，被调用方是否标注为必填？（调用方漏传必填字段 → P1）
- [ ] **枚举值对齐**：传入枚举字段时，调用方传入的所有枚举值均在被调用方合法枚举范围内，无多无少
- [ ] **分页参数对齐**：分页接口的参数名（`page`/`pageNo`/`pageNum`、`size`/`pageSize`/`limit`）和含义一致
- [ ] **时间格式对齐**：时间字段的格式明确约定（`ISO 8601` / `timestamp` / `yyyy-MM-dd HH:mm:ss`）

**B. 响应体字段对齐**

- [ ] **外层结构对齐**：响应体顶层包装结构一致（如 `{code, message, data}` vs `{status, msg, result}` 须统一）
- [ ] **data 层字段名对齐**：调用方解析使用的每个字段名，在被调用方响应体 `data` 对象中真实存在
- [ ] **列表/分页结构对齐**：分页响应中 `total`/`totalCount`/`count`、`list`/`records`/`items` 字段名对齐
- [ ] **嵌套对象字段对齐**：响应体存在嵌套对象时，嵌套层级和字段名均对齐
- [ ] **空值处理对齐**：字段为空时，被调用方返回 `null` 还是省略该字段？调用方是否做了对应空值处理
- [ ] **枚举响应值对齐**：响应中的枚举字段，调用方的展示逻辑覆盖了被调用方返回的所有枚举值

**C. 通用约定对齐**

- [ ] **命名风格统一**：同一调用关系中字段命名风格统一（统一驼峰或统一下划线）；混用时有明确序列化转换说明
- [ ] **时区处理统一**：双端对时间字段的时区处理一致（统一 UTC 或统一本地时间）
- [ ] **错误码对齐**：被调用方定义的所有错误码，调用方均有对应的错误处理分支

**输出格式（发现问题时）**：
```
⚠️ [前端→后端] POST /api/order/submit 字段不对齐（P1）：
   请求字段：
   - 前端传 `userId`(number)，后端定义 `user_id`(string) → 命名+类型均不对齐
   - 前端传 `orderItems`，后端定义 `items` → 字段名不对齐
   响应字段：
   - 前端解析 `data.orderId`，后端返回 `data.order_id` → 字段名不对齐
   - 前端 status 枚举处理含 `CANCELLED`，后端枚举仅定义 `CANCEL` → 枚举值不一致
```
