# 就绪检查 + 通用质量检查清单

> Step 0 后加载，全流程保留。包含就绪检查清单和通用文档质量检查清单。

## 零、就绪检查清单（Readiness Check）

在正式评审前执行前置检查。**任意一项不满足，直接返回"文档未就绪"，不进行评审**：

- [ ] 文档是否有明确的标题和版本号
- [ ] 文档是否有作者和日期信息
- [ ] 文档是否包含背景/目标章节（说明"为什么做"）
- [ ] 文档主体内容是否完整（非空白占位符）
- [ ] 异常流程/错误处理描述是否存在（占比 > 5%）

> **测试用例评审专属替换规则**：当评审类型为「测试用例评审」时，第3项替换为「文档是否包含测试范围说明」，第4项替换为「用例格式是否完整（ID/描述/输入/预期/结果）」，第5项替换为「异常场景用例占比 > 20%」。

> **详细设计评审专属替换规则**：当评审类型为「详细设计评审」时，第3项替换为「文档是否包含模块职责和接口清单概述」，第5项替换为「异常处理/降级策略描述占比 > 10%」。

就绪检查不通过时输出：

```
⚠️ 文档未就绪，无法进行评审

缺失项：
- [列出具体缺失的检查项]

建议：请补充以上内容后重新提交评审。
```

## 五、通用文档质量检查清单

### 5.1 文档质量检查

- [ ] **版本管理**：文档是否有版本号和变更历史？
- [ ] **术语一致**：文档中的术语是否前后一致？是否有术语表？
- [ ] **图表清晰**：流程图、时序图、架构图是否清晰易读？
- [ ] **无占位符**：文档中是否有未填写的占位符（如：TODO、TBD、待定）？
- [ ] **链接有效**：文档中的外部链接是否有效？

### 5.2 跨文档一致性检查（全流程评审时）

- [ ] **需求→架构**：架构中的每个技术决策是否能追溯到需求文档中的具体条目？
- [ ] **架构→详细设计**：详细设计中的每个模块是否在架构文档中有对应定义？
- [ ] **详细设计→测试**：测试用例中的每个场景是否能追溯到详细设计中的接口或业务规则？
- [ ] **需求→测试**：测试用例是否覆盖了需求文档中的所有AC？
