# 需求评审检查清单

> Step 2 需求评审时加载。包含价值背景、逻辑闭环、AC完整性、一致性、逆向流程、可测试性检查。

## 一、需求评审检查清单

### 1.1 价值与背景检查

- [ ] **背景说明**：文档是否说明了"为什么要做这个功能"？是否有明确的用户痛点或业务价值描述？
- [ ] **目标用户**：是否明确了目标用户群体？用户画像是否具体？
- [ ] **成功指标**：是否定义了可量化的成功指标（如：DAU提升X%、转化率提升X%）？
- [ ] **优先级依据**：为什么现在做这个功能？是否有优先级排序依据？

**What-If 追问：**
> "如果这个功能不做，用户会怎样？" — 如果答不上来，说明价值不清晰。

---

### 1.2 逻辑闭环检查

- [ ] **主流程完整**：正常使用路径是否完整描述（从入口到出口）？
- [ ] **异常流程**：每个步骤的失败情况是否有处理方案？
- [ ] **边界条件**：数值边界、时间边界、状态边界是否明确？
- [ ] **并发场景**：多用户同时操作同一资源时，系统行为是否定义？
- [ ] **网络中断**：操作过程中网络中断，系统状态如何？数据是否安全？
- [ ] **第三方不可用**：依赖的外部服务不可用时，用户体验是什么？
- [ ] **状态机完整**：如果有状态流转，是否覆盖了所有状态和转换条件？

**What-If 追问清单：**
> - "如果用户在步骤X中断网络了，系统状态应该是什么？"
> - "如果两个用户同时操作同一条数据，会发生什么？"
> - "如果第三方支付回调延迟了，系统会怎么处理？"
> - "如果用户重复提交同一个请求，会产生重复数据吗？"

---

### 1.3 验收标准（AC）完整性检查

- [ ] **AC存在性**：每个功能点是否都有对应的验收标准？
- [ ] **AC可量化**：AC是否使用了具体数字或可验证的条件？（"系统要很快"❌ → "P99响应时间 < 500ms"✅）
- [ ] **AC可测试**：每条AC是否可以被测试用例直接覆盖？
- [ ] **AC无歧义**：AC的描述是否存在多种解读方式？

**常见不合格AC示例（需要标记为P1）：**
- "系统响应要快" → 缺少量化指标
- "界面要美观" → 无法验证
- "支持大量用户" → 无具体数字
- "数据要准确" → 无精度定义

---

### 1.4 需求一致性检查

- [ ] **内部一致**：同一文档内的需求是否存在矛盾？
- [ ] **跨需求一致**：与其他已有需求是否存在冲突？
- [ ] **技术约束一致**：需求是否与已知的技术约束冲突（如：要求实时但架构是批处理）？
- [ ] **优先级一致**：需求优先级是否与业务目标一致？

---

### 1.5 逆向流程检查

| 正向操作 | 对应逆向操作 | 是否已定义 |
|---------|-----------|-----------|
| 创建/新增 | 删除/撤销 | □ |
| 提交/发布 | 撤回/下架 | □ |
| 支付/扣款 | 退款/退款失败处理 | □ |
| 绑定/关联 | 解绑/解除关联 | □ |
| 审批通过 | 审批驳回/撤销审批 | □ |
| 锁定/冻结 | 解锁/解冻 | □ |

---

### 1.6 可测试性检查

- [ ] **无模糊描述**：文档中是否存在"尽量"、"适当"、"合理"等无法验证的词汇？
- [ ] **有明确输入输出**：每个功能的输入条件和预期输出是否明确？
- [ ] **有错误码定义**：异常情况下的错误提示/错误码是否定义？
- [ ] **有数据格式定义**：关键字段的格式、长度、类型是否明确？
