# 术语表

> 本文件定义 task-decomposer 技能中使用的专业术语和缩写，供 AI 在任务拆解时保持术语一致性。

---

## 核心术语

| 术语 | 含义 | 使用场景 |
|-----|------|---------|
| **PRD** | 产品需求文档（Product Requirements Document） | Step 1 输入源 |
| **SAD** | 软件架构设计文档（Software Architecture Document） | Step 1 输入源 |
| **任务拆解** | 将大功能分解为可独立开发的最小任务单元 | 全流程核心目标 |
| **规则表** | `doc/detailed/项目规则.md`，记录全局约定和契约 | Step 5 输出产物 |
| **详设文档** | 后端/前端/小程序的详细设计文档，按模块/页面域拆分 | Step 3/4 输出产物 |
| **LC 规则** | Lifecycle Contract，生命周期契约规则（如 LC-001） | 项目规则.md 锚点 |
| **原子任务** | 不可再拆分的最小开发单元（1个接口/1个页面/1个组件） | Step 3 拆解粒度 |
| **依赖关系** | 任务间的前置条件（如"登录接口"必须先于"用户信息接口"） | Step 2 排序依据 |
| **端** | 系统的一个独立用户入口（后端/Web前端/小程序） | Step 0 端类型识别 |
| **功能链** | 一个完整业务操作所需的全部接口集合（含隐含支撑接口） | Step 2.5 推导目标 |

## 禁用词汇对照表

> 以下左列词汇禁止出现在详设文档中，应替换为右列的规范表达。

| 禁用词汇 | 替代表达 |
|---------|---------|
| "实现登录功能" | ✅ "POST /api/auth/login 用户登录接口" |
| "做一个用户列表页面" | ✅ "GET /api/users + UserList.vue 用户列表页" |
| "优化性能" | ❌ 必须明确优化目标（如"优化用户列表接口响应时间至 < 200ms"） |
| "后续处理" | ❌ 必须明确处理时机和负责人 |
| "参考 XX 项目" | ❌ 必须明确复用哪些代码/组件 |
| "类似功能" | ❌ 必须明确差异点和新增逻辑 |
| "简单实现" | ❌ 必须列出具体实现步骤 |
| "调用接口" | ✅ "调用 POST /api/orders 创建订单接口" |
| "保存数据" | ✅ "INSERT INTO orders 表，字段：order_id/user_id/amount" |
| "显示信息" | ✅ "展示字段：用户名/订单号/创建时间，UI 布局：卡片列表" |

## 规则表术语（LC 规则）

### 规则编号规范
| 前缀 | 含义 | 示例 |
|-----|------|------|
| **LC-BE-** | 后端契约规则 | LC-BE-001 RESTful 接口规范 |
| **LC-FE-** | Web 前端契约规则 | LC-FE-001 Vue3 组件命名规范 |
| **LC-MP-** | 小程序契约规则 | LC-MP-001 WXML 四件套规范 |
| **LC-DB-** | 数据库契约规则 | LC-DB-001 表命名规范 |
| **LC-SEC-** | 安全契约规则 | LC-SEC-001 JWT 认证流程 |
| **LC-COM-** | 通用契约规则 | LC-COM-001 Git 提交规范 |

### 常见规则类型
| 规则类型 | 说明 | 示例 |
|---------|------|------|
| **技术栈** | 明确框架版本 | "后端使用 Spring Boot 3.2 + Java 17" |
| **目录结构** | 代码组织方式 | "Controller/Service/Repository 三层架构" |
| **命名规范** | 变量/函数/文件命名 | "接口路径统一 kebab-case：`/api/user-orders`" |
| **错误处理** | 异常和错误码规范 | "业务异常返回 400 + 错误码 E1001" |
| **日志规范** | 日志级别和格式 | "ERROR 级别记录异常堆栈，INFO 记录关键业务节点" |

## 设计明细术语

### 后端明细字段
| 字段名 | 含义 | 示例 |
|-------|------|------|
| **接口路径** | RESTful API 路径 | `POST /api/orders` |
| **请求参数** | 请求体字段 | `{ userId, productId, quantity }` |
| **响应数据** | 响应体字段 | `{ orderId, status, createdAt }` |
| **业务逻辑** | 核心处理步骤（伪代码） | "1. 校验库存 2. 扣减库存 3. 创建订单" |
| **数据库操作** | SQL 语句类型和表名 | `INSERT INTO orders`, `UPDATE inventory` |
| **依赖接口** | 前置调用的其他接口 | "依赖 GET /api/products/:id 查询商品信息" |

### 前端明细字段
| 字段名 | 含义 | 示例 |
|-------|------|------|
| **页面路径** | 路由路径 | `/orders/detail/:id` |
| **组件名称** | Vue/React 组件文件名 | `OrderDetail.vue` |
| **调用接口** | 依赖的后端接口 | `GET /api/orders/:id` |
| **展示字段** | UI 展示的数据字段 | "订单号、商品名、数量、总价、状态" |
| **交互逻辑** | 用户操作和响应 | "点击【取消订单】按钮 → 调用 DELETE /api/orders/:id" |
| **公共组件** | 复用的组件库 | "AppButton、AppCard、AppTag" |

## 任务拆解粒度

| 粒度级别 | 定义 | 工作量估算 | 示例 |
|---------|------|-----------|------|
| **史诗（Epic）** | 大功能模块 | 2-4 周 | "订单管理模块" |
| **特性（Feature）** | 独立功能点 | 3-5 天 | "创建订单功能" |
| **任务（Task）** | 可独立开发的原子单元 | 0.5-1 天 | "POST /api/orders 创建订单接口" |
| **子任务（Sub-task）** | 任务的实现细节 | 1-2 小时 | "OrderService.createOrder 业务逻辑" |

**task-decomposer 输出粒度**：**Task 级别**（原子任务），每个任务对应 1 个接口或 1 个页面/组件。

## 端类型术语

| 端类型 | 技术栈 | 输出文件示例 |
|-------|--------|------------|
| **后端** | Spring Boot / Django / Express / Gin | `用户_认证.md`、`订单_支付.md` |
| **Web 前端** | Vue 3 / React / Next.js | `前端_订单管理.md`、`前端_首页.md` |
| **微信小程序** | WXML/WXSS/JS/JSON 四件套 | `小程序_首页.md`、`小程序_订单.md` |
| **多端并存** | 后端 + 多个前端 | 每个端各自独立文档 |

## 依赖关系术语

| 关系类型 | 说明 | 示例 |
|---------|------|------|
| **强依赖** | 必须先完成 A 才能开始 B | "创建订单接口" → "订单列表接口" |
| **弱依赖** | 可并行开发，但联调时需 A 已完成 | "用户登录接口" 与 "商品列表接口" |
| **数据依赖** | B 需要 A 产生的数据 | "订单详情接口" 依赖 "订单创建接口" 产生的订单ID |
| **接口依赖** | 前端页面依赖后端接口 | "订单列表页" 依赖 "GET /api/orders" |

## 质量检查术语

| 检查项 | 说明 | 合格标准 |
|-------|------|---------|
| **完整性** | 所有 PRD/SAD 功能点都有对应接口/页面 | 需求覆盖率 100% |
| **无歧义** | 任务描述清晰可执行 | 开发人员无需反复询问 |
| **可测试** | 任务有明确的验收标准 | 每个接口有 Given/When/Then |
| **粒度适中** | 任务工作量 0.5-1 天 | 不拆成过小的子任务 |
| **依赖清晰** | 任务间依赖关系明确 | 开发顺序无歧义 |
