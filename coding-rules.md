# AI 编码铁律 — 系统强制执行

## 零号铁律：禁止直接修改文件

任何涉及文件修改的操作（包括但不限于改端口号、改配置、修文案、增删代码），
**必须先委托给 coding-executor agent，再由它走完三阶段流程才能编辑**。
严禁以"小修改"、" trivial "、" 简单配置 " 等任何理由跳过此流程。

系统已配置权限：直接编辑会弹出确认提示。如果看到确认弹窗，说明你正在违规——立即停止，重新委托给 coding-executor。

## coding-executor 三阶段流程

### 阶段一：加载上下文 + 编码前验证

```
1. 调 memory_init_session() + memory_search_summaries()
   → 查看进行中任务和历史记录，了解项目上下文
2. bash scripts/gate.sh pre <模块> <文档>
   → 验证 doc/ 下设计文档是否存在，不通过不准编码
```

### 阶段二：读文档 + 编码

```
3. 读 doc/detailed/ 下对应模块的详细设计文档
   → 重点：业务规则、API 契约、实体定义、DDL
4. 严格按文档编码，不越界、不补位、不自由发挥
```

### 阶段三：后置验证 + 保存记忆

```
5. 五维核对 + bash scripts/gate.sh post <模块> '<核对报告>'
   → 任一维度未确认就 exit 1，必须修复重跑
6. 验证通过后调 memory_save_summary() 记录本次编码
   → 自动继承 verify-coding.sh post 输出的字段模板
```

### 五维核对

| # | 核对项 | 你要做什么 |
|---|--------|----------|
| ① | 业务逻辑对齐 | 代码行为与文档伪代码一致？ |
| ② | 接口 URL 对齐 | 路由路径、HTTP 方法与 API 契约一致？ |
| ③ | 请求/响应参数对齐 | 字段名、类型、结构层次与契约一致？ |
| ④ | 实体 ↔ 表结构对齐 | Entity 字段与 DDL 脚本一一映射？ |
| ⑤ | 无漂移代码 | 有没有写文档里没有的东西？ |

### 端不对齐处理

```
发现端与端数据不一致
  └─→ 查 doc/ 下 API 契约和详细设计文档
       ├─ 文档有定义 → 以文档为准，改偏离的那一端
       └─ 文档没定义 → 停下来，报告用户
                        （输出路径：差异详情 + "文档未定义，请确认"）
```

---

## 一句话总结

**先加载记忆 → 跑脚本读文档 → 按文档写代码 → 跑脚本验代码 → 保存记忆。脚本没通过 = 工作没完成。**
